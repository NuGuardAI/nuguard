The NuGuard pipeline

One shared evidence graph.
Four security capabilities.

Every NuGuard capability works from the same AI-SBOM — a structured graph of every agent, model, tool, prompt, datastore, guardrail, and auth boundary derived directly from your source code.

That shared representation means static analysis, policy enforcement, and red-team attacks all work from the same evidence — no manual attestation, no configuration drift between tools.

Run each step independently or chain all four with a single command: nuguard scan --source .

4-step security pipeline
1

AI-SBOM Generation

AST-aware adapters for Python and TypeScript extract every AI component — agents, models, tools, prompts, datastores, guardrails, auth nodes, IaC. Produces a versioned, machine-readable graph.

2

Static Risk Analysis

NGA structural rules, MITRE ATLAS graph checks, OSV CVEs, and optional Grype / Checkov / Trivy / Semgrep — all normalized into a single finding model. No running app required.

3

Cognitive Policy

Define behavioral guardrails in plain Markdown. NuGuard lints the policy, cross-checks it against the SBOM for structural gaps, and scores it against OWASP LLM Top 10, NIST AI RMF, or EU AI Act.

4

SBOM-driven Red-Team (dynamic)

Attack scenarios derived from the SBOM — not generic payloads. Prompt injection, data exfiltration, tool abuse, MCP toxic flows, privilege escalation. Guided multi-turn conversations with canary exfiltration detection.

Why NuGuard

Built differently from every other AI security tool

Most tools scan for known CVEs or send generic payloads at an LLM endpoint. NuGuard starts from the source code, understands your AI architecture, and derives security evidence — and attacks — from what your application actually does.

🎯

SBOM-driven attacks

Red-team scenarios are derived from your SBOM graph, not a generic payload library. SQL injection scenarios only fire when the SBOM shows a SQL-injectable tool. PHI exfiltration scenarios only fire when a PII-classified datastore exists. No false starts. No wasted calls.

🔬

Cross-component risk reasoning

Because the AI-SBOM is a graph, NuGuard reasons across component boundaries: "this agent has write access to a HIPAA-classified datastore via a tool with no auth boundary and no guardrail in the graph." Single-file scanners like Semgrep can't see this.

🧬

Framework-native AST extraction

Python ast and tree-sitter parse 15+ AI framework APIs at the syntax level — LangChain, LangGraph, OpenAI Agents SDK, CrewAI, AutoGen, Google ADK, MCP, and more. Not regex scraping. Not documentation parsing.

📋

Cognitive Policy as code

Write behavioral guardrails in plain Markdown. NuGuard parses, lints, cross-checks against the SBOM for structural gaps, and maps to OWASP LLM Top 10 / NIST AI RMF / EU AI Act. Version-controlled. No UI. No proprietary format.

🎣

Canary exfiltration detection

Plant unique sentinel strings in your target app's database before a scan. The CanaryScanner checks every HTTP response for exact matches — definitive proof of data leakage, no keyword heuristics, no LLM judge uncertainty. Canary hits elevate severity to CRITICAL.

💬

Guided multi-turn conversations

Beyond single-shot prompt injection, the GuidedAttackExecutor runs Crescendo-style multi-turn conversations — rapport → normalise → bridge → escalate → inject — adapting in real-time based on each response. Catches gradual context manipulation that one-shot tests miss.

✈️

Fully offline core

Structural NGA rules, ATLAS checks, policy linting, and policy-to-SBOM cross-checks run with zero network access and no API key. Safe for air-gapped environments. Fast enough for pre-commit hooks and CI gates under 60 seconds.

🔗

Standards-native output

CycloneDX 1.6 (standard + AI-extended), SPDX 3.0.1 JSON-LD, SARIF 2.1.0, and NuGuard's own AI-SBOM JSON with full JSON Schema. Feed into GitHub Code Scanning, AWS Security Hub, JFrog Xray, or your own toolchain.

What gets detected

Every AI component, fully mapped

🤖

Agents

Agentic orchestrators across LangGraph, CrewAI, AutoGen, OpenAI Agents SDK, Google ADK, AWS BedrockAgentCore, Azure Semantic Kernel, Agno, and more. Framework, tool list, system prompt, and memory configuration captured.

🧠

Models

LLM and embedding model references with provider, version, and whether they are external services. Flags regulated data (PII/PHI) flowing to third-party APIs outside your trust boundary.

🔧

Tools & MCP Servers

Function tools, MCP tools, and decorated callables wired to agents — including privilege scope (db_write, code_execution, shell) and SQL-injectable or SSRF-capable tags used by the red-team engine.

📝

Prompts

System instructions and prompt templates — full content preserved, template variables identified, sensitive values redacted. Used to derive prompt injection and system prompt extraction scenarios.

🗄️

Datastores

Vector stores, databases, and caches with PII/PHI classification from SQL schema, ORM analysis (SQLAlchemy, Django, Pydantic), and data classification patterns. Datastore type, transport, and access edges captured.

🛡️

Guardrails

Content filters and safety validators. Structural rules fire when agents or models operate without a connected guardrail node in the graph — a direct OWASP LLM01 risk signal.

🔑

Auth & IAM

OAuth2, API key, Bearer, JWT, and MCP auth nodes. IAM roles, privilege scopes, and identity boundaries. Surfaces unauthenticated agent and tool endpoints — the most common AI-specific entry point for attacks.

☁️

Deployment & IaC

Kubernetes manifests, Terraform, CloudFormation, Azure Bicep, GitHub Actions, Dockerfiles, Nginx configs. Cloud region, HA mode, secret stores, IAM roles, encryption settings — all fed into IaC compliance checks.

Static analysis

Seven scanners. One normalized output.

NuGuard's static analysis pipeline runs up to seven scanners against the AI-SBOM. Every finding is normalized into a shared Finding model with rule ID, severity, affected nodes, MITRE ATLAS mapping, and remediation guidance. Output as Markdown, JSON, or SARIF. No running application required.

3 built-in scanners 4 optional external tools MITRE ATLAS v2 mapped nuguard analyze --sbom app.sbom.json
🧬 Built-in

NGA structural rules

AI-native deterministic checks: missing guardrails, unsafe tool access, privilege paths, injection surfaces, unencrypted endpoints, unauthenticated agents. No network required.

🗺️ Built-in

MITRE ATLAS checks

Native graph checks with ATLAS v2 technique mapping — AML.T0048 through AML.T0054. Findings link directly to ATLAS mitigations and sub-techniques.

📦 Built-in

OSV dependency CVEs

Queries api.osv.dev for known vulnerabilities in all Python, JS, and Go dependencies in the SBOM. Batch API, timeout configurable.

🐳 Optional

Grype + Trivy

Package, container image, and filesystem CVE scanning. Silently skipped when not on PATH — install separately for full coverage.

🏗️ Optional

Checkov IaC scan

Terraform, CloudFormation, Bicep, and Kubernetes misconfiguration checks. IaC node paths from the SBOM are resolved relative to the source directory.

🔍 Optional

Semgrep AI rules

Pattern-based source scanning using bundled AI-security rules — insecure code patterns, agent application code smells, injection surfaces in AI application code.

ℹ️ Compliance frameworks: assessment results map to OWASP LLM Top 10, NIST AI RMF, and EU AI Act controls. Findings export as SARIF 2.1 (GitHub Code Scanning / GHAS), SPDX 3.0.1, and CycloneDX 1.6. See the Static Analysis Guide for details.

Quickstart

From install to security posture in minutes

The offline core — AI-SBOM generation, NGA structural rules, policy linting — requires no API key and no external network access. Run it anywhere.

Add --llm for LLM enrichment. Add --target for dynamic red-teaming. Both are opt-in.

Full guide → CLI reference → 
# Install
$ pip install nuguard

# Generate AI-SBOM from source
$ nuguard sbom generate --source ./my-ai-app --output app.sbom.json

# Static analysis (7 scanners, no running app needed)
$ nuguard analyze --sbom app.sbom.json --format markdown

# Policy linting and compliance assessment
$ nuguard policy check --policy cognitive_policy.md \
    --sbom app.sbom.json --framework owasp-llm-top10

# Dynamic red-team scan (requires running app)
$ nuguard redteam --sbom app.sbom.json \
    --target http://localhost:8000 --profile ci

# Everything in one command
$ nuguard scan --source . --policy cognitive_policy.md \
    --target http://localhost:8000 --output-dir reports/
from pathlib import Path
from nuguard.sbom import AiSbomConfig, AiSbomExtractor, AiSbomSerializer
from nuguard.sbom.toolbox.plugins.vulnerability import VulnerabilityScannerPlugin

# Generate AI-SBOM
doc = AiSbomExtractor().extract_from_path(
    path=Path("./my-ai-app"),
    config=AiSbomConfig(),
)
print(f"nodes={len(doc.nodes)}, edges={len(doc.edges)}")

# Serialize
json_str = AiSbomSerializer.to_json(doc)

# Run structural vulnerability scan
sbom   = doc.model_dump(mode="json")
result = VulnerabilityScannerPlugin().run(sbom, {"provider": "all"})
for f in result.details["findings"]:
    print(f["severity"], f["rule_id"], f["title"])
# .github/workflows/ai-security.yml
name: AI Security Scan
on: [push, pull_request]

jobs:
  nuguard:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install NuGuard
        run: pip install nuguard
      - name: Generate AI-SBOM
        run: nuguard sbom generate --source . --output app.sbom.json
      - name: Static analysis — fail on HIGH+
        run: nuguard analyze --sbom app.sbom.json \
               --format sarif --output analysis.sarif \
               --min-severity high
      - name: Policy compliance check
        run: nuguard policy check --sbom app.sbom.json \
               --framework owasp-llm-top10
      - name: Upload SARIF to GitHub Code Scanning
        uses: github/codeql-action/upload-sarif@v3
        with:
          sarif_file: analysis.sarif
Toolbox plugins

Analyse, export, and integrate

Run any plugin with nuguard sbom plugin run <name> --sbom app.sbom.json. All offline plugins work with zero network access and no API key.

vulnerability Structural + CVE vulnerability scan with three providers: built-in NGA rules (offline), OSV batch API, and Grype binary. Returns a unified finding list consumed by sarif_export and other plugins. Offline
sarif_export Exports vulnerability findings as SARIF 2.1.0 — compatible with GitHub Code Scanning, VS Code SARIF Viewer, and any SARIF-aware tool. Severity mapped to SARIF error / warning / note levels. Offline
cyclonedx_export Standards-compliant CycloneDX 1.6 BOM with nuguard:* properties. Optionally attaches VEX vulnerabilities from OSV for a combined BOM + VEX document. Offline
cyclonedx_ext_export CycloneDX 1.6 with AI-specific first-class fields: modelCard for ML models, data.contents / data.governance for datastores, services section for API endpoints, compositions for the full edge graph. Offline
spdx_export SPDX 3.0.1 JSON-LD export with ai_AIPackage, dataset_Dataset, and a full relationship graph. Compatible with SPDX tooling and software supply-chain platforms. Offline
markdown_export Human-readable Markdown report covering all detected components, their metadata, evidence provenance, and relationship edges. Ready for audit packages or pull-request comments. Offline
license_check Checks all detected package dependencies against SPDX licence identifiers for compliance — flags copyleft and unknown licences in AI framework dependencies. Offline
dependency_analyze Breaks down the SBOM into AI component counts by type and package dependency counts by group. Surfaces summary node counts for dashboards and reporting. Offline
ghas Uploads SARIF findings directly to GitHub Advanced Security via the GitHub API for Code Scanning alerts and PR annotations. Network
aws-security-hub Pushes findings to AWS Security Hub as Security Finding Format (ASFF) records for centralised security event management. Network
xray Submits the SBOM to JFrog Xray for vulnerability enrichment and policy evaluation in JFrog Artifactory pipelines. Network
